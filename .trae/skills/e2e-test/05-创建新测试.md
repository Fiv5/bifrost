## 创建新测试

### 步骤 1: 确定测试类型

| 类型 | 说明 | 创建方式 |
| --- | --- | --- |
| 规则测试 | 测试代理规则行为 | 在 `e2e-tests/rules/` 下按模块功能和测试目标创建 `.txt` 规则文件，并在 `e2e-tests/tests/` 下创建配套脚本 |
| Admin API 测试，独立 E2E 测试 | 测试管理接口/自管理服务的完整测试 | 在 `e2e-tests/tests/` 目录下创建脚本 |


### 步骤 2: 创建规则文件（规则测试）

规则相关测试必须遵守以下结构约束：

1. 规则文件统一存放在 `e2e-tests/rules/`
2. 先按模块功能创建或复用子目录，例如 `forwarding/`、`request_modify/`、`response_modify/`
3. 再按测试目标拆分 `.txt` 文件，一个文件只承载一个清晰的测试目标
4. 文件顶部先写“测试目标说明”，再写规则内容
5. 配套执行脚本必须存放在 `e2e-tests/tests/`

推荐目录示例：

```text
e2e-tests/
├── rules/
│   └── request_modify/
│       └── headers.txt
└── tests/
    └── test_request_headers.sh
```

在对应分类目录下创建 `.txt` 规则文件：

```bash
# 模块目录示例: forwarding, request_modify, response_modify, redirect, priority, control, template
cat > e2e-tests/rules/your_category/your_test.txt << 'EOF'
# 测试目标: 验证某类规则在指定输入下的匹配与改写行为
# 场景说明: 说明这个文件覆盖的单一目标，避免混入无关语义

pattern.test.com protocol://target
EOF
```

如果一个文件开始同时覆盖多个目标，例如“请求头改写 + Cookie 改写 + Body 改写”，应拆成多个 `.txt` 文件，而不是继续堆在一起。

### 步骤 3: 创建测试脚本

规则测试和 Admin API 测试脚本统一放在 `e2e-tests/tests/`。使用 `test_utils/` 中的工具库：

```bash
#!/bin/bash
set -uo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
ROOT_DIR="$(cd "$SCRIPT_DIR/.." && pwd)"
source "$ROOT_DIR/test_utils/assert.sh"
source "$ROOT_DIR/test_utils/http_client.sh"

PROXY_PORT="${PROXY_PORT:-18888}"
PROXY="http://127.0.0.1:${PROXY_PORT}"

passed=0
failed=0

test_your_feature() {
    local test_url="https://your-feature.test.com/api/test"
    https_request "$test_url"

    if assert_status_2xx "$HTTP_STATUS" "Your feature test"; then
        _log_pass "Your feature test passed"
        ((passed++))
    else
        _log_fail "Your feature test failed"
        ((failed++))
    fi
}

main() {
    echo "=========================================="
    echo "  Your Feature E2E Tests"
    echo "=========================================="

    test_your_feature

    echo ""
    echo "Results: $passed passed, $failed failed"
    [ $failed -eq 0 ]
}

main "$@"
```

脚本文件命名建议直接体现测试目标，例如 `test_header_replace.sh`、`test_rules_hot_reload.sh`。若脚本依赖某个具体规则文件，脚本中应明确引用对应的 `e2e-tests/rules/.../*.txt` 路径。

### 步骤 4: 更新覆盖率文档

新增测试后更新 `rules/COVERAGE.md`。

### 验证准则

1. 测试必须通过：所有相关测试用例必须通过
2. 测试独立性：测试之间不应有依赖关系
3. 清理资源：使用 trap 确保测试结束后清理资源
4. 边界条件：包括空值、特殊字符、大数据等边界情况
5. 断言描述：断言描述应清晰说明测试目的
6. 目录约束：规则文件只能放在 `e2e-tests/rules/`，配套脚本只能放在 `e2e-tests/tests/`
7. 文件可读性：规则文件顶部必须包含测试目标说明，帮助后续维护者快速判断覆盖范围
