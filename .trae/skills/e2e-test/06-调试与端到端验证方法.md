## 调试、端到端验证方法(最佳方式)

构造测试用例，进行端到端测试（API 验证，和交互验证），覆盖 HTTP/1.1、HTTP/2、HTTPS、SOCKS5、CONNECT-UDP 等场景，覆盖 TLS 与非 TLS 情况，覆盖 TLS 解包和不解包场景，覆盖 HTTP/3 场景。

所有管理端接口都必须是 `http://127.0.0.1:{port}/_bifrost/api/` 开头。

```bash
# 先编译，再启动代理服务
CARGO_TARGET_DIR=./.bifrost-ui-target cargo build --bin bifrost
RUST_LOG=debug BIFROST_DATA_DIR=./.bifrost-test ./.bifrost-ui-target/debug/bifrost start -p 8890 --unsafe-ssl

# 查看代理转发的请求详情
curl -s --proxy http://127.0.0.1:8890 http://test.com/api | jq .

# 使用 curl 调用管理端接口
curl -s -X POST http://127.0.0.1:8890/_bifrost/api/replay/execute -H "Content-Type: application/json" -d '{"request":{"method":"GET","url":"https://httpbin.org/get","headers":[["Accept","*/*"]]},"rule_config":{"mode":"enabled"}}'

# 观察所有的日志：代理日志、管理端日志、接口日志、代理请求日志
# 必须使用 chrome mcp工具打开 http://127.0.0.1:8890/_bifrost/ 验证交互功能

# 特别说明提高验证效率的方法
# 如果仅仅改动了 web 页面功能，不涉及管理端接口，那么只需要验证 web 页面功能是否正常即可。则启动代理服务后，可以启用前端 dev 模式，devserver 会将 `/_bifrost/api`、`/_bifrost/public` 与 `/_bifrost/ws` 代理到目标代理服务端口。
# 示例：
# pnpm -C web dev -- --backend-port 8890
# 然后访问：http://127.0.0.1:3000/_bifrost/
```

## 必须遵守的调试顺序

1. 先确认不是旧进程
   - `pgrep -af 'bifrost start|cargo run --bin bifrost'`
   - `lsof -nP -iTCP:{port} -sTCP:LISTEN`
2. 再确认新进程真的 ready
   - `curl -sS http://127.0.0.1:{port}/_bifrost/api/proxy/address`
3. 再做 API 与 UI 验证
   - 先看 `/_bifrost/api/traffic`、`/_bifrost/api/traffic/{id}`
   - 再看页面列表、详情、Messages 面板
4. UI 与 API 不一致时，必须抓 websocket frame
   - 重点看 `/api/push` 的 query、`framesent`、`framereceived`
   - 典型用途：判断页面是否真的发出了 `need_traffic=true` 订阅

## 最小化排查方法

遇到实时推送问题时，不要一上来跑整套 UI。优先做最小链路验证：

1. 用最小脚本造一条请求，例如 HTTP、SSE、WebSocket、CONNECT
2. 轮询 `/_bifrost/api/traffic` 找到对应 `id`
3. 用 `/_bifrost/api/traffic/{id}` 确认最终落库状态
4. 同时抓页面 `/api/push` 消息，确认是否收到了 `insert/update`

如果 API 正确、push 也正确，但页面不更新，优先怀疑前端 store 的消费/替换条件，而不是数据库。
