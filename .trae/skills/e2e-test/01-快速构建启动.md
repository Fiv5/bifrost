## 快速构建启动（推荐）

目标：仅构建后端并快速启动代理服务。凡是涉及管理端静态资源、push 订阅、Traffic 页面行为的验证，必须先完成编译，再启动新进程，不要复用旧进程做结论。

首先需要复制 `~/.bifrost/certs/` 到启动目录，确保启动时可以加载到客户端证书。例如`.bifrost-e2e-test`目录。

```bash
# 推荐：先编译，再用编译产物启动
CARGO_TARGET_DIR=./.bifrost-ui-target cargo build --bin bifrost
BIFROST_DATA_DIR=./.bifrost-e2e-test ./.bifrost-ui-target/debug/bifrost start -p 8890 --unsafe-ssl
```

等待构建完成，并启动服务。必须完成以下确认后再测试：
```bash
# 1. 监听进程正确
lsof -nP -iTCP:8890 -sTCP:LISTEN

# 2. 管理端已 ready
curl -sS http://127.0.0.1:8890/_bifrost/api/proxy/address
```

如果启动命令进入 CA 证书交互界面，先显式选择并让进程进入 `Proxy is up and running` 状态，再继续后续操作。

## 前端 UI 快速启动与验证（dev 模式）

目标：本地快速验证管理端 UI；前端 devserver 会将 `/_bifrost/api` 与 `/_bifrost/ws` 代理到目标代理服务端口。

```bash
# 启动代理（示例端口 8890）
export BIFROST_DATA_DIR=./.bifrost-e2e-test
rm -rf "$BIFROST_DATA_DIR"
CARGO_TARGET_DIR=./.bifrost-ui-target cargo build --bin bifrost
./.bifrost-ui-target/debug/bifrost start -p 8890 --unsafe-ssl

# 启动前端 devserver，并将代理目标指向 8890
pnpm -C web dev -- --backend-port 8890
```

验证：

1. 打开：`http://127.0.0.1:3000/_bifrost/`
2. 确认页面可正常加载、列表有数据/可操作
3. 如需确认代理是否生效，可打开浏览器 Network，检查请求是否走 `/_bifrost/api` 路径且返回正常
4. 如需验证 push 订阅，必须抓浏览器 websocket：
   `page.on("websocket", ws => { ws.on("framesent", ...); ws.on("framereceived", ...); })`
   用于确认页面是否真的向服务端订阅了 `need_traffic=true`
