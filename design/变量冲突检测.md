# 变量冲突检测

## 背景

当用户为同一域名启用了多套规则集时，不同规则集可能通过 markdown 代码块（如 ` ```data `）定义了同名的内联变量，但变量内容不同。由于变量在合并时采用"先到先得"策略（`entry(k).or_insert(v)`），后面规则集中的同名变量会被静默忽略，导致用户设置的规则不生效却无法感知原因。

## 需求

在灵动岛（Dynamic Island）组件中检测并展示跨规则集的变量冲突，帮助用户快速发现和定位问题。

## 实现方案

### 后端

扩展 `GET /api/rules/active-summary` 接口，在现有响应中增加 `variable_conflicts` 字段：

```rust
#[derive(Debug, Serialize)]
struct VariableConflict {
    variable_name: String,
    definitions: Vec<VariableDefinition>,
}

#[derive(Debug, Serialize)]
struct VariableDefinition {
    rule_name: String,
    group_id: Option<String>,
    value_preview: String, // 截取前 80 字符
}

#[derive(Debug, Serialize)]
struct ActiveSummaryResponse {
    total: usize,
    rules: Vec<ActiveRuleItem>,
    variable_conflicts: Vec<VariableConflict>,
}
```

检测逻辑：遍历所有已启用的规则文件，对每个规则文件调用 `extract_markdown_value_blocks` 提取内联变量，记录每个变量名对应的（规则文件名, 变量值）列表。当同一变量名在不同规则文件中有不同值时，标记为冲突。

### 前端

在 `RulesDynamicIsland.tsx` 中：

1. 扩展 `ActiveSummaryResponse` 和 `ActiveRuleItem` 类型定义
2. 当存在变量冲突时，在灵动岛 Badge 上显示警告图标
3. 展开面板中在规则列表上方显示冲突警告区域

## 依赖

- `bifrost_core::rule::parser::extract_markdown_value_blocks`（需要导出为公共函数）
- `bifrost_storage::RulesStorage`

## 测试方案

### 单元测试

- `test_detect_variable_conflicts_same_name_different_values`：两个规则文件定义 `data` 变量但值不同，返回冲突
- `test_no_conflicts_when_values_match`：两个规则文件定义 `data` 变量且值相同，不返回冲突
- `test_no_conflicts_single_rule_file`：单个规则文件不产生冲突

### E2E 测试

- 创建两个规则集分别定义 `data` 变量但内容不同，调用 active-summary API 验证 `variable_conflicts` 非空
- 验证冲突信息包含正确的规则文件名和值预览

### 真实场景测试

- 启动服务后在 Web UI 创建多个规则集使用相同变量名但不同值，观察灵动岛是否展示冲突提示
